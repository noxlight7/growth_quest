# Growth Quest

`growth_quest` - это Django + React прототип для создания и прохождения текстовых приключений с LLM-генерацией сюжета.

В текущей конкурсной рамке проект развивается как proof-of-concept принципа благотворного влияния генеративных сюжетов игр будущего на личность игрока. Игра должна оставаться игрой: с сюжетом, NPC, выбором, последствиями, доверием и возможностью исправить вред.

Люди уже играют, и игровые миры уже влияют на их способы действия и реагирования. По мере внедрения генеративного ИИ сюжеты, реакции NPC и последствия будут все чаще создаваться на лету. Проект исследует, как направлять эту генерацию так, чтобы влияние было психологически и нравственно благотворным.

Выключение явного нравственного модуля не делает генеративную игру нейтральной: сюжет, награды и последствия все равно влияют на игрока, но основания этого влияния становятся менее прозрачными. При этом проект не предполагает единственной универсальной реализации "правильного": люди и сообщества различаются в нравственных представлениях, желаемых целях личностного роста и приемлемых психологических подходах.

## Конкурсный контекст

Проект готовится для `Designing the Future / GCD4F 2026` в треке `AI for Education`.

Для конкурса важно показать не "учебную платформу с AI", а более сильную идею:

- будущие игры будут всё чаще генерировать сюжет, реакции NPC, квесты и последствия на лету;
- без явных ограничений такие системы будут оптимизировать в первую очередь драму, новизну и удержание;
- `growth_quest` показывает, как в такой игровой цикл можно встроить два независимо включаемых модуля без скрытого психопрофилирования и без превращения игры в назидание.

## Что это за продукт

Проект стоит читать не как конечный продукт, а как текстовый proof-of-concept для любых будущих генеративных игр.

Его рабочая логика такая:

- игрок действует внутри истории, а не проходит анкету;
- сюжет отвечает последствиями через мир, события, отношения и доступные варианты;
- значимые поступки и изменения мира сохраняются как сюжетные события, чтобы будущие сцены могли правдоподобно учитывать память NPC, доверие, слухи и доступные возможности;
- модуль нравственных причинно-следственных связей (кармы) делает добрые, честные и ответственные линии сюжета более содержательными, а жестокие и эгоистичные решения связывает с долгосрочными издержками;
- ошибка не означает "провал личности", а может открывать repair arc;
- evidence собирается по наблюдаемым игровым действиям и репликам;
- модуль психологически поддерживающей сюжетной адаптации использует наблюдаемые действия и органичные возможности для empathy, cooperation, self-regulation, responsible decisions, restorative action, help-seeking и inclusion;
- оба модуля включаются независимо и опционально по этическим причинам;
- целевой продуктовый эффект состоит в том, чтобы игрок сам хотел включить модули, потому что с ними игровой процесс интереснее, полнее и насыщеннее;
- текущий GM-интерфейс нужен для настройки и отладки прототипа, но не является центром принципа.

В следующей архитектурной итерации игрок должен явно выбирать собственные цели роста, а не получать приписанные системой "проблемы". Предполагается поддержка модулей из разных доверенных источников с описанными ценностными основаниями и психологическими подходами. Перед применением каждый модуль должен пройти всестороннюю оценку влияния, включая проверку психологической безопасности, риска манипуляции, предвзятости и качества сюжетных последствий.

Проект не должен превращаться в:

- обязательную образовательную среду;
- закрытую оценку личности игрока;
- психометрический или "диагностический" инструмент;

## Что уже реализовано

- редактор шаблонов приключений: локации, расы, персонажи, фракции, системы, техники, события и дополнительная информация;
- запуск `runs` из шаблонов;
- hero setup перед стартом;
- lobby и multiplayer turn flow;
- игровой экран с историей, вводом действий, генерацией следующего хода, `pass`, rollback, regenerate и экспортом истории в PDF;
- импорт и экспорт шаблонов в JSON;
- создание переведённой копии шаблона через LLM с пакетной обработкой карточек;
- модерация и публикация шаблонов;
- JWT-аутентификация, вход по username или email;
- growth objectives на уровне сценария;
- редактируемые narrative constraints для growth-layer внутри prompt-building;
- opt-in LLM-анализ наблюдаемых игровых действий без keyword-based классификатора;
- отдельный модуль нравственных причинно-следственных связей (кармы) со связями событий с персонажами, локациями и фракциями;
- независимые переключатели growth-analysis и долговременных сюжетных последствий на уровне шаблона и запущенной игры;
- optional debrief / in-world reflection;
- accessibility profile, player profile и GM dashboard;
- portfolio export для анализа proof-of-concept;
- safety baseline и локальный запуск без обязательного платного LLM;
- отдельный `story_locale` (`en`, `ru`, `zh-CN`) для общего языка истории независимо от языка интерфейса игрока.

## Архитектурная рамка

Проект остается `modular monolith`, а не переписывается в набор микросервисов.

Стек:

- backend: Django, Django REST Framework, Channels;
- frontend: React;
- database: PostgreSQL;
- LLM providers: `local`, `openai-compatible`, `ollama`, `yandex`.

Логические контуры в коде уже разделены:

- story loop и генерация ответа;
- модуль психологически поддерживающей сюжетной адаптации: narrative constraints, отдельный LLM-вызов для observable evidence и органичных repair opportunities;
- модуль нравственных причинно-следственных связей (кармы): отдельные таблицы значимых событий и связей с персонажами, локациями и фракциями;
- safety review;
- GM dashboard.

Это важно для конкурсной демонстрации: один и тот же AI-модуль не должен одновременно писать историю, оценивать игрока, модерировать риск и собирать отчет.

Язык интерфейса хранится в персональном accessibility profile игрока. Язык истории задается отдельно
для шаблона и общего multiplayer run через `story_locale`. Narrative prompt локализуется на этот язык,
а скрытые prompts для structured analysis и JSON-схемы остаются на английском. Для конкурсного demo
используется `story_locale=en`.

Долговременная память не добавляется в prompt целиком. Для текущей сцены выбирается не более 12 активных событий: сначала события, связанные с текущей локацией, персонажами партии и их фракциями, затем более важные и свежие. Вместе с событием передается только один уровень связанных сущностей; рекурсивное добавление других событий этих сущностей запрещено. Во время сворачивания старых сообщений модель также может обновлять, закрывать и архивировать эту память.

## Ограничения и принципы MVP

- Главный экран продукта - это игра, а не дашборд.
- Целевая аудитория принципа - все игроки будущих генеративных игр.
- Система анализирует наблюдаемое поведение в истории, а не "внутреннюю сущность" игрока.
- Персонализация будущей психологически поддерживающей адаптации должна опираться на явно выбранные игроком цели роста, а не на скрытый вывод о его недостатках.
- Reflection допустима только как мягкий и по возможности in-world слой, а не как обязательный опрос после каждого хода.
- Choice cards нужны для доступности и снижения cognitive load, а не как набор "правильных ответов".
- Нравственный и социальный рост в MVP должен проявляться через игровой процесс: последствия, выбор, отношения, доверие и repair opportunities.
- Конструктивные линии сюжета должны в среднем ощущаться богаче и приятнее, чем жестокие или циничные, даже если последние иногда дают краткосрочную выгоду.
- При недоступности или некорректном ответе модели turn-analysis ничего не синтезирует по ключевым словам и не блокирует игровой цикл.
- Оба дополнительных сюжетных модуля выключены по умолчанию и независимо включаются создателем сценария или GM для конкретного run.
- Safety baseline остается отдельным консервативным контуром.
- Подключаемые в будущем модули разных подходов не заменяют safety baseline и не должны применяться без прозрачного описания источника, версии и предварительной оценки влияния.

## Быстрый старт через Docker

1. Скопируйте корневой env-файл:

```bash
cp .env.example .env
```

2. Поднимите сервисы:

```bash
docker compose up --build
```

После запуска:

- frontend: `http://localhost:3000`
- backend: `http://localhost:8000`
- PostgreSQL из Docker: `127.0.0.1:5433`

Контейнер backend при старте сам выполняет `migrate` и `collectstatic`.
Контейнер `db_backup` создает резервные копии базы в `backups/`.

Для production-like конфигурации можно использовать:

```bash
cp .env.production.example .env.production
ENV_FILE=.env.production docker compose up --build
```

## Локальная разработка

### Backend

1. Подготовьте `.env` в корне репозитория.
2. Убедитесь, что PostgreSQL доступен по параметрам из `.env`.

Если вы хотите использовать БД из `docker compose`, оставьте `POSTGRES_HOST=localhost` и переключите `POSTGRES_PORT` на `5433`.

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r backend/requirements.txt

cd backend
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

Backend будет доступен на `http://localhost:8000`.

### Frontend

```bash
cd frontend
cp .env.example .env
npm install
npm start
```

Frontend будет доступен на `http://localhost:3000`.

## Конфигурация окружения

Основные настройки берутся из корневого `.env`.

- `DJANGO_SECRET_KEY`, `DJANGO_DEBUG`, `DJANGO_ALLOWED_HOSTS`, `ALLOWED_ORIGINS` - базовая конфигурация backend;
- `POSTGRES_*` - подключение к PostgreSQL;
- `REACT_APP_API_URL` - адрес API для frontend;
- `CHANNEL_REDIS_URL` - Redis для Channels, нужен в production;
- `LLM_PROVIDER`, `LLM_MODEL`, `LLM_BASE_URL`, `LLM_API_KEY`, `OLLAMA_URL`, `OLLAMA_MODEL`, `YC_*` - выбор и настройка LLM provider;
- `TEMPLATE_TRANSLATION_BATCH_MAX_ITEMS`, `TEMPLATE_TRANSLATION_BATCH_MAX_CHARS` - лимиты пакета карточек для перевода копии шаблона через LLM;
- `HISTORY_MAX_PROMPT_POSTS`, `HISTORY_TAIL_UPDATE_POSTS`, `HISTORY_UPDATE_MAX_TOKENS`, `HISTORY_UPDATE_STRICT_MAX_TOKENS` - ограничения на историю и update prompts.

Создание переведённой копии требует реального LLM provider. Локальный `LLM_PROVIDER=local`
предназначен для разработки без внешнего ключа и не выполняет перевод.

## Конкурсное демо

Проект можно запускать без внешнего AI-ключа:

```bash
cd backend
LLM_PROVIDER=local python manage.py runserver
```

Для конкурсного видео используется отдельный развлекательный фэнтезийный сценарий:

```bash
cd backend
python manage.py seed_last_beacon_demo
```

По умолчанию команда создает demo-логин:

- username: `demo_host`
- password: `demo12345`

Сценарий `The Last Beacon Pass` показывает работу модулей внутри обычного фэнтезийного
приключения: психологически поддерживающие возможности и долговременные нравственные
причинно-следственные связи должны оставаться частью сюжета, а не внешним назиданием.
Seed-команда сразу публикует шаблон, чтобы его можно было запустить из списка опубликованных
приключений.

## Основные экраны

- `/` - список шаблонов, запусков и опубликованных приключений;
- `/adventures/:id/edit` - редактор шаблона;
- `/adventures/runs/:id/edit` - редактор уже созданного run;
- `/adventures/:id/lobby` - lobby перед запуском;
- `/adventures/:id/hero` - настройка героя;
- `/adventures/:id/play` - игровой экран;
- `/adventures/:id/gm` - панель GM;
- `/moderation` - очередь модерации и публикации;
- `/admin` - Django admin.

Legacy routes `/adventures/:id/teacher` и `/adventures/:id/facilitator` перенаправляются на `/adventures/:id/gm`.

## Текущее позиционирование

Если коротко, цель проекта такая:

`growth_quest` должен оставаться демонстрацией того, как будущие AI-generated game worlds могут сохранять игровую природу и при этом встраивать психологически поддерживающие возможности и нравственные причинно-следственные связи в саму драматургию. Главный результат проекта - не конкретная текстовая игра, а переносимый принцип для игр будущего.
